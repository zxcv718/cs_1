이번 과제의 평가 항목을 기준으로 설명드리겠습니다. 먼저 기능 동작 검증 항목에서는, 단순히 명령을 한 번 실행한 것이 아니라 실제 수행 흔적과 결과를 모두 남겼다는 점을 보여드릴 수 있습니다. 터미널 기본 조작 부분에서는 현재 위치 확인, 숨김 파일 포함 목록 확인, 디렉토리 생성, 디렉토리 이동, 빈 파일 생성, 파일 내용 확인, 복사, 이동 및 이름 변경, 삭제까지를 로그로 남겼습니다. 권한 실습에서는 파일과 디렉토리 각각에 대해 변경 전후 권한을 비교해 두었습니다. Docker 점검에서는 `docker --version`과 `docker info`를 함께 기록해 명령어 설치와 엔진 동작 여부를 분리해서 확인했습니다. `hello-world` 실행 성공, 이미지와 컨테이너 목록 확인, `docker logs`, `docker stats` 같은 운영 명령 흔적도 따로 남겼습니다. 커스텀 이미지 부분에서는 Dockerfile로 `codyssey-web:1.0` 이미지를 실제로 빌드하고, 컨테이너 실행과 포트 매핑 결과를 로그와 브라우저 캡처로 증명했습니다. 볼륨 부분에서는 컨테이너 삭제 전후에 같은 데이터가 유지되는 과정을 로그로 확인했고, Git 설정과 GitHub 및 VS Code 연동도 로그와 스크린샷으로 남겼습니다.

다음으로 동작 구조 설계 항목에서는, 제가 프로젝트 디렉토리 구조를 어떤 기준으로 나눴는지 설명할 수 있습니다. README는 전체 흐름과 검증 문서의 중심 역할을 하도록 두었고, Dockerfile은 이미지 빌드 규칙, `site/`는 웹 콘텐츠 원본, `docs/logs/`는 CLI 기반 증거, `docs/screenshots/`는 브라우저와 VS Code 화면 증거를 모으는 위치로 분리했습니다. 이렇게 나눈 이유는 코드와 증거와 설명을 한 폴더에 섞지 않고 역할별로 분리해야 평가자가 README만 보고도 결과물 위치를 쉽게 추적할 수 있기 때문입니다. 포트와 볼륨 설정 역시 README 재현 섹션에 단계별 명령으로 정리해 두었습니다. 포트 매핑은 어떤 포트가 호스트 포트이고 어떤 포트가 컨테이너 포트인지 설명했고, 바인드 마운트는 왜 호스트 파일 수정이 즉시 반영되는지, 볼륨은 왜 컨테이너 삭제 후에도 데이터가 남는지 명령과 결과를 연결해 설명할 수 있습니다.

핵심 기술 원리 적용 항목에서는 이미지와 컨테이너의 차이를 빌드, 실행, 변경 관점에서 설명할 수 있습니다. 이미지는 Dockerfile을 기반으로 만들어지는 실행 템플릿이고, 컨테이너는 그 이미지를 실제로 실행한 인스턴스입니다. 그래서 이미지가 바뀌려면 다시 빌드가 필요하지만, 실행 중인 컨테이너는 그 이미지를 기반으로 동작하는 별도의 런타임 결과물입니다. 바인드 마운트 사례에서는 8080과 8081이 왜 다른 결과를 보여 줄 수 있는지도 이 관점으로 설명 가능합니다. 8080은 빌드 당시 이미지 안에 복사된 정적 파일을 보고 있고, 8081은 호스트의 `site/index.html`을 직접 연결해 현재 파일 상태를 즉시 반영합니다. 포트 매핑이 필요한 이유도 설명할 수 있습니다. 컨테이너 내부 80번 포트는 호스트 브라우저가 직접 접근할 수 없기 때문에, `-p 8080:80`처럼 호스트 포트와 컨테이너 포트를 연결해야 합니다. 절대 경로와 상대 경로는 바인드 마운트와 파일 작업에서 왜 중요한지 README에 설명되어 있고, 권한 숫자 표기인 755와 644도 `r=4`, `w=2`, `x=1` 규칙으로 해석하는 방식까지 문서화했습니다.

심층 인터뷰 항목에 대해서도 설명 가능합니다. 예를 들어 호스트 포트가 이미 사용 중이라 포트 매핑이 실패하는 경우라면, 먼저 `docker ps`로 같은 포트를 쓰는 컨테이너가 있는지 확인하고, 그다음 `lsof -i :8080`처럼 호스트 포트를 점유 중인 프로세스를 확인한 뒤, 점유 대상을 정리하거나 다른 포트 번호로 바꾸는 순서로 접근하겠다고 답할 수 있습니다. 컨테이너 삭제 후 데이터가 사라진 경험이 있다면, 그 원인이 컨테이너 내부 파일시스템에만 데이터를 두었기 때문인지 먼저 확인하고, 이후 Docker 볼륨이나 바인드 마운트처럼 컨테이너 수명과 분리된 저장 경로를 사용해야 한다고 설명할 수 있습니다. 가장 어려웠던 지점과 해결 과정도 트러블슈팅 항목에 정리해 두었습니다. `docker info` 실패, OrbStack이 `damaged`로 보이던 문제, `xcodebuild` 경로 오류 같은 실제 사례에 대해 문제, 원인 가설, 확인, 해결, 배운 점을 순서대로 남겨 두었기 때문에, 단순히 문제가 있었다는 수준이 아니라 어떻게 판단하고 해결했는지까지 설명할 수 있습니다.

정리하면, 이 제출물은 단순한 명령 실행 모음이 아니라 각 평가 항목에 맞춰 기능 동작 증거, 구조 설계 의도, 핵심 기술 원리, 면접형 질문 대응 근거를 모두 연결해 둔 상태입니다. 그래서 README만 읽어도 무엇을 했는지, 왜 그렇게 했는지, 어떤 로그와 스크린샷이 근거인지, 그리고 평가 질문이 들어왔을 때 어떤 논리로 답할 수 있는지가 한 흐름으로 이어지도록 구성했습니다.
